Главные термины GitHub понятным языком 
  • Репозиторий (Repository / Repo) — это папка вашего проекта. Она хранится в облаке GitHub и содержит все файлы, картинки и историю их изменений. 
  • Коммит (Commit) — это сохранение проекта. Похоже на сохранение в компьютерной игре. Вы сделали часть работы, зафиксировали её коммитом и снабдили короткой записью (например: «Добавил кнопку оплаты»). Если что-то сломается, можно легко вернуться к этому сохранению. 
  • Пуш (Push) — отправка изменений с вашего компьютера на сайт GitHub. Пока вы не сделали «пуш», все ваши коммиты остаются только у вас на жестком диске. 
  • Бранч (Branch / Ветка) — это копия проекта внутри того же репозитория для безопасных экспериментов. Главная ветка обычно называется main или master. Если вы хотите протестировать новую функцию, вы создаете новую ветку (копию), меняете файлы там, а когда все заработает — объединяете её с главной веткой. 
Ответ на вопрос про папки 
Да, группировать файлы по папкам можно и нужно.
GitHub полностью повторяет структуру папок вашего компьютера. Вы можете создавать любую вложенность: assets/images/, src/components/ и так далее. 
⚠️ Важный нюанс: Git (система, на которой работает GitHub) не умеет сохранять пустые папки. Чтобы папка отобразилась на сайте, внутри неё должен лежать хотя бы один файл (например, пустой текстовый файл .gitkeep). 
Пошаговый мануал: как начать работу (без командной строки) 
Для новичков проще всего использовать официальную программу GitHub Desktop или делать все прямо через сайт. Вот инструкция для сайта: 
  1. Создайте репозиторий: Нажмите кнопку New (или плюс в правом верхнем углу) на сайте GitHub. Введите имя проекта и нажмите Create repository. 
  2. Загрузите файлы: Нажмите кнопку Upload files. Перетащите мышкой ваши папки и файлы в окно браузера. 
  3. Сделайте коммит: Внизу страницы в поле Commit changes напишите, что именно вы добавили (например: «Первый запуск сайта»). Нажмите зеленую кнопку Commit changes. 
  4. Изменяйте код: Вы можете открывать файлы прямо на сайте, нажимать на иконку карандаша, редактировать текст и снова нажимать Commit changes. 
Вы сказали: "Я пока собираюсь пользоваться веб-интерфейсом. Как в этом случае разложить файлы, уже хранящиеся на GitHub, по папкам?"
Разложить уже загруженные файлы по папкам прямо в веб-интерфейсе GitHub можно двумя способами. Git устроен так, что «переместить» файл — это значит переименовать путь к нему. 
Вот два самых простых варианта, как это сделать без сторонних программ. 
Способ 1: Через встроенный веб-редактор (Самый удобный) 
В GitHub встроен полноценный редактор кода, похожий на обычную программу. Он открывается прямо в браузере. 
  1. Откройте ваш репозиторий на сайте GitHub. 
  2. Нажмите на клавиатуре клавишу . (точку). 
  3. Страница перезагрузится, и откроется редактор (Visual Studio Code для браузера). 
  4. В левой панели вы увидите дерево ваших файлов:
    • Чтобы создать папку, нажмите на иконку папки с плюсом (New Folder). Назовите её.
    • Чтобы переместить файл, просто перетащите его мышкой в нужную папку. 
  5. Перейдите на вкладку Source Control в левом вертикальном меню (иконка ветления с кружком, третья сверху). 
  6. В поле Message напишите, что вы сделали (например: Разложил файлы по папкам). 
  7. Нажмите синюю кнопку Commit & Push (или галочку сверху). 
  8. Закройте вкладку редактора и обновите страницу репозитория — все файлы на месте. 
Способ 2: Ручное переименование файла (Для 1-2 файлов) 
Если нужно переместить всего один файл, это можно сделать через стандартную страницу GitHub. 
  1. Нажмите на файл, который хотите переместить. 
  2. В правом верхнем углу окна с кодом нажмите на иконку карандаша (Edit this file). 
  3. Посмотрите на строку над кодом, где написано имя файла. Нажмите мышкой перед именем файла. 
  4. Введите название новой папки и поставьте косую черту / (слэш).
    • Пример: было script.js, вы написали js/. Текст автоматически превратится в папку, а имя файла сместится правее: js / script.js. 
  5. Прокрутите страницу в самый куз, в поле Commit changes напишите комментарий к изменению. 
  6. Нажмите зеленую кнопку Commit changes. Файл окажется в новой папке. 
Вы сказали: "Файлов у меня не много, поэтому способ с переименованием пути предпочтительней. А как создать браннч для экспериментов? И что такое Pull Request, issue и codespace?"
Как создать бранч (ветку) на сайте GitHub 
Создать ветку для экспериментов прямо в браузере можно буквально в два клика: 
  1. Откройте главную страницу вашего репозитория.
  2. В левой части (чуть выше списка файлов) найдите кнопку с выпадающим списком, где написано main или master (рядом с иконкой разветвления).
  3. Нажмите на неё и в появившейся строке поиска введите название вашей новой ветки (например: test-design или experiment).
  4. Под строкой появится надпись: Create branch: [название_ветки] from 'main'. Нажмите на неё. 
Готово! GitHub переключит вас в эту ветку. Теперь любые изменения, удаления или создание папок, которые вы делаете, будут происходить только внутри этой ветки, не ломая главный проект. 
Что такое Pull Request, Issue и Codespace? 
Эти три инструмента — сердце GitHub, которые делают его удобным для работы. 
1. Pull Request (Пулл-реквест / PR) 
Это запрос на объединение веток.
Когда вы поработали в своей экспериментальной ветке (experiment) и поняли, что всё отлично работает, вам нужно перенести эти изменения в главную ветку (main). 
  • Вы нажимаете кнопку Contribute -> Open Pull Request.
  • GitHub показывает вам и вашим коллегам наглядную разницу: что было изменено, добавлено или удалено.
  • Если всё в порядке, вы нажимаете кнопку Merge Pull Request (Слить). Изменения из экспериментальной ветки перетекают в главную. 
2. Issue (Ишью / Задачи) 
Это встроенный список задач, багов и идей для проекта. Выглядит как форум или доска объявлений внутри репозитория. 
  • Если в коде нашлась ошибка или появилась идея (например: «Поменять цвет фона на синий»), вы создаете New Issue.
  • Там можно вести обсуждение, прикреплять скриншоты и назначать ответственных.
  • Когда задача выполнена, её закрывают (Close issue). Это помогает не держать всё в голове. 
3. Codespace (Кодспейс) 
Это готовый виртуальный компьютер для программирования, который запускается прямо в облаке GitHub. 
  • Вам не нужно устанавливать на свой личный компьютер сложные программы, редакторы кода или базы данных.
  • Вы нажимаете одну кнопку, и GitHub в новой вкладке браузера собирает полноценную рабочую среду (интерфейс программы VS Code), где вы можете писать код, запускать его и тестировать.
  • Для личных небольших проектов это бесплатно (в пределах бесплатных часов в месяц). 
Вы сказали: "Ну вот я переместил один файл в созданную тут же на лету папку. Как создать Pull Request? Как я понял, Pull Request - это слить ветку для экспериментов с основной. Но я же еще не создавал ни одной ветки кроме главной."
Как это должно было выглядеть, если делать «по правилам» через Pull Request: 
Чтобы в будущем опробовать этот процесс, алгоритм действий должен быть следующим: 
  1. Создать ветку: Сначала вы нажимаете на кнопку main, вводите имя новой ветки (например, reorganize-files) и создаете её. GitHub автоматически переключает вас в неё. 
  2. Сделать изменения: Находясь в ветке reorganize-files, вы переименовываете путь к файлу (создавая папку) и нажимаете Commit changes. 
  3. Создать Pull Request: После сохранения коммита сам GitHub сверху покажет желтую плашку с предложением: «reorganize-files had recent pushes less than a minute ago. Compare & pull request». 
  4. Вы нажимаете на эту кнопку, проверяете, что файлы переместились правильно, и подтверждаете слияние с веткой main. 
Что делать сейчас? 
Вы уже навели порядок в главной ветке, и это абсолютно нормально для личного проекта, когда вы работаете один. Ветки и Pull Request чаще всего нужны, когда: 
  • Проект большой, и вы боитесь случайно что-то сломать в работающей версии.
  • Вы работаете в команде и хотите, чтобы коллеги сначала проверили ваш код (сделали Code Review), прежде чем он попадет в главный проект. 
Хотите сейчас попробовать создать пустую тестовую ветку, внести в неё любое мелкое изменение (например, создать пустой текстовый файл) и пройти весь путь создания и одобрения Pull Request, чтобы увидеть, как это работает на практике? 
Вы сказали: "У меня на странице, где сравниваются изменения, нет кнопки Merge Pull Request. Есть вот то что на скрине"
Всё верно, вы находитесь на шаге №1 — создании запроса. Кнопка Merge (слияние) появится только после того, как вы выполните это действие. 
Вот что нужно сделать прямо сейчас, глядя на ваш скриншот: 
1. Выберите тип Pull Request 
В выпадающем меню у вас есть два варианта. Для обычного проекта выбирайте первый: 
  • Create pull request (выбран синим) — стандартный запрос. Он сразу сообщает всем, что код готов к проверке и объединению.
  • Create draft pull request (ниже) — черновик. Нужен, если работа еще не закончена, но вы хотите показать код коллегам для совета. 
2. Нажмите на зеленую кнопку 
Просто кликните по самой зеленой кнопке Create pull request. 
Что произойдет дальше? 
Как только вы нажмете эту кнопку, страница обновится. GitHub создаст пулл-реквест и автоматически запустит проверку (проверит, нет ли конфликтов между ветками). 
И вот тогда на новой странице вы увидите ту самую долгожданную большую зеленую кнопку Merge pull request. Вам останется только нажать её и подтвердить слияние. 
Хотите нажать кнопку прямо сейчас и прислать скриншот того, что изменилось на странице, чтобы мы завершили слияние веток вместе? 
Вы сказали: "У меня зависло на этапе Checking for the ability to merge automatically... Соответственно и кнопка Merge pull request не активна. Это надо будет смотреть в uBlock Origin, что он там заблокировал на странице, мне кажется в этом все дело. Ну в целом я понял как это все делается. спасибо. А там рядом с кнопкой Merge pull request написано You can also merge this with the command line. Это наверно доступно только если установить приложение?"
Да, вы абсолютно правы — зависание на плашке Checking for the ability to merge automatically... очень часто связано с блокировщиками рекламы вроде uBlock Origin, встроенной защитой браузера (например, в Brave или Firefox) или временным сбоем скриптов самого GitHub. Странице просто не удается получить ответ от сервера. Вы отлично во всем разобрались! 
Что касается надписи «You can also merge this with the command line»: 
Нет, устанавливать официальное приложение GitHub Desktop для этого не обязательно. 
Эта надпись означает, что слияние можно провести через обычную командную строку (терминал) вашего компьютера с помощью стандартной консольной утилиты Git. 
Вот как это работает глобально: 
  1. Утилита Git — это базовая программа без графического интерфейса. Она устанавливается на компьютер разработчика. 
  2. Если нажать на эту надпись на сайте, GitHub покажет точные текстовые команды, которые нужно по очереди скопировать и вставить в терминал. 
  3. Эти команды заставят ваш компьютер скачать экспериментальную ветку, объединить её с главной прямо на вашем жестком диске и отправить (сделать push) результат обратно на сайт. 
Поскольку вы пока планируете работать только через веб-интерфейс, эту надпись можно просто игнорировать. Как только вы разберетесь с uBlock, кнопка Merge pull request на сайте станет активной, и командная строка вам не понадобится.
