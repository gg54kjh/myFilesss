﹡Как проверить утечки памяти в Firefox
Чтобы настройки, включая тональный режим (Tone), не обнулялись во время замены батареек, используйте простой лайфхак: 
  • Меняйте батарейки по одной. Не вынимайте весь старый комплект сразу. Замените сначала один элемент питания, затем второй и третий.
  • Соблюдайте скорость. Телефон способен удерживать настройки в памяти за счет остаточного заряда конденсаторов в течение примерно 15–30 секунд после извлечения батареек. Если уложиться в это время, часы и настройки не сбросятся. 
Вы сказали: "А вот замена батареек по одной - там тоже надо успеть заменить все 4 батарейки за эти 30 секунд?"
Нет, если вы меняете батарейки по одной, то торопиться и укладываться в 30 секунд не нужно. 
Вся суть этого метода в том, что в телефоне постоянно остается питание: 
  • Когда вы вынимаете первую батарейку, остальные три продолжают питать микросхему памяти.
  • Вы спокойно вставляете вместо нее новую. Питание восстанавливается.
  • Затем вы так же не спеша меняете вторую, третью и четвертую. 
Главное — не вытаскивать две или более батарейки одновременно. Меняйте их строго по очереди: вытащили одну — сразу вставили новую. При таком подходе настройки и время гарантированно не сбросятся, даже если вся замена займет у вас пару минут. 
Вы сказали: "Нет, вопросов по этой теме больше нет. Есть другой вопрос. Имеется js код:const str = 'https://www.somedomain.com/abc/sdf/?tl=en';const a = document.createElement('a'); a.href = str;const newUrl = a.origin + a.pathname + 'my_string'; // в дальнейшем к переменной a я больше не обращаюсь, а имею дело только с newUrlСозданную ссылку я даже не приаттачиваю к DOM-дереву и соответственно не удаляю когда она уже не нужна. Не будет ли происходить утечка оперативы из-за того, что где-то болтается ссылка на эту переменную a?"
Нет, никакой утечки оперативной памяти в данном случае не произойдет. 
Сборщик мусора (Garbage Collector) в современных JavaScript-движках работает по алгоритму Mark-and-Sweep (маркировка и очистка). Он удаляет объекты не потому, что они «не приаттачены к DOM», а потому, что они становятся недостижимыми (unreachable) из глобального контекста выполнения. 
Как это сработает в вашем коде 
  1. Вы объявили переменную a через const в определенной области видимости (например, внутри функции или блока).
  2. Вы создали элемент в памяти, записали из него данные в newUrl.
  3. Как только выполнение кода выходит за рамки области видимости, где была объявлена const a, ссылка на этот элемент теряется.
  4. Сборщик мусора видит, что объект <a> в памяти больше никем не используется (на него никто не ссылается из активного JavaScript-кода и он не находится в дереве DOM), и автоматически удаляет его во время следующего цикла очистки. 
Единственное исключение (когда утечка возможна) 
Утечка памяти (так называемый Detached DOM node — изолированный узел DOM) могла бы возникнуть только в одном случае: если бы вы сохранили глобальную ссылку на переменную a. 
Например, если бы вы написали window.mySavedLink = a; или случайно создали замыкание (closure), удерживающее эту переменную внутри какой-то долгоживущей функции или обработчика событий. Но поскольку вы написали, что в дальнейшем к переменной a вообще не обращаетесь, память очистится абсолютно корректно. 
Альтернативный совет: Если вам нужно просто распарсить URL без создания «тяжелых» объектов интерфейса вроде элементов DOM, в современном JavaScript (начиная с ES6) гораздо удобнее и чище использовать встроенный класс URL: 
javascript
const str = 'https://www.somedomain.com/abc/sdf/?tl=en';
const urlObj = new URL(str);
const newUrl = urlObj.origin + urlObj.pathname + 'my_string';
Это работает быстрее, не задействует механизмы рендеринга браузера и семантически точнее описывает вашу задачу. 
"Да, функция с этим кодом вызывается довольно часто. А насчёт Performance/Memory для Chrome - я пользуюсь Firefox. Там есть этот функционал?"
Да, в Firefox есть отличный инструмент для этого — вкладка Память (Memory) в Инструментах разработчика. 
Как проверить утечки памяти в Firefox 
  1. Откройте Инструменты разработчика (F12 или Ctrl + Shift + I / Cmd + Option + I).
  2. Перейдите на вкладку Память (Memory). Если её нет, включите её в настройках панели (шестерёнка).
  3. Нажмите кнопку Сделать снимок (Take snapshot) — это зафиксирует текущее состояние памяти (кучи / heap).
  4. Выполните действие на сайте, которое часто вызывает вашу функцию (например, проскролльте страницу или покликайте).
  5. Сделайте второй снимок памяти.
  6. Переключите режим отображения с «Дерево вызовов» на «Сравнение» (Aggregate / Compare) и выберите два ваших снимка. Вы увидите, какие объекты выросли в количестве. Если ваша функция создавала и не удаляла объекты, вы увидите рост HTMLAnchorElement или URL. 
Важный нюанс при частом вызове (Garbage Collection) 
Поскольку функция вызывается часто, в случае с document.createElement('a') браузеру приходится на каждый вызов создавать тяжелый объект DOM-узла со всеми его скрытыми свойствами, а затем сборщику мусора приходится тратить ресурсы на их постоянное удаление. 
Это не вызовет утечку памяти (она не будет расти бесконечно), но может вызвать «дребезг» памяти (GC churn) — кратковременные микрофризы интерфейса, когда сборщик мусора резко включается, чтобы вычистить сотни накопившихся объектов. 
Решение: Если вы перейдете на предложенный ранее new URL(str), нагрузка снизится. Но если функция вызывается экстремально часто (например, по событию scroll или mousemove), оптимально вынести создание объекта за пределы функции, чтобы использовать его повторно: 
javascript
// Объект создается один раз при инициализации скрипта
const urlParser = new URL('https://temporary.placeholder'); 

function yourFrequentFunction(str) {
    // Просто обновляем свойство href у существующего объекта
    urlParser.href = str; 
    const newUrl = urlParser.origin + urlParser.pathname + 'my_string';
    // ... дальнейшая логика
}
При таком подходе браузер вообще не будет создавать новые объекты в памяти при каждом вызове функции, и нагрузка на Garbage Collector упадет до нуля.

  • Проверить вашу функцию на оптимальность работы с другими строками
  • Посмотреть, как дебажить частые вызовы через вкладку «Производительность» 
Вы сказали: "Вот ещё такой код. Я использую его чтобы декодировать HTML сущности. Тут можно найти альтернативу созданию textarea по аналогии с объектом URL? const textarea = document.createElement('textarea'); textarea.innerHTML = someHTML_code; const decodedText = textarea.value;"
Да, отличный вопрос! Для этой задачи тоже есть современная и гораздо более производительная альтернатива, которая не задействует тяжелые механизмы рендеринга DOM-элементов браузера. 
В современном JavaScript для этого используется встроенный интерфейс DOMParser. Он работает со строками напрямую в памяти и идеально подходит для частого вызова. 
Альтернативный вариант с DOMParser 
javascript
const parser = new DOMParser();
const doc = parser.parseFromString(someHTML_code, 'text/html');
const decodedText = doc.documentElement.textContent;
Как оптимизировать для частых вызовов (без создания объектов) 
По аналогии с примером про URL, если эта функция вызывается очень часто, создавать new DOMParser() внутри нее при каждом вызове — неэффективно. Лучше вынести экземпляр парсера за пределы функции: 
javascript
// Создаем парсер один раз при инициализации
const htmlParser = new DOMParser();

function decodeHtmlEntities(someHTML_code) {
    // Используем один и тот же экземпляр повторно
    const doc = htmlParser.parseFromString(someHTML_code, 'text/html');
    return doc.documentElement.textContent;
}
Почему это лучше вашего текущего кода? 
  1. Производительность: document.createElement('textarea') заставляет браузер создавать полноценный элемент интерфейса со множеством скрытых свойств, стилей и событий, даже если он не добавлен на страницу. DOMParser просто обрабатывает строку как текст. 
  2. Безопасность: Использование свойства innerHTML (как в вашем коде с textarea) потенциально опасно. Если в someHTML_code случайно попадет вредоносный скрипт, в определенных условиях это может привести к XSS-уязвимостям. DOMParser изолирует парсинг в отдельном документе памяти, что гораздо безопаснее. 
Если вы хотите заменять базовые HTML-сущности на чистом JavaScript через регулярные выражения (без использования DOMParser или textarea), то сниппет должен выглядеть совершенно иначе. 
  • Плюс: Это самый быстрый способ в JavaScript. Он работает в разы быстрее, чем DOMParser, так как вообще не задействует логику браузера по разбору документа. Идеально для экстремально частых вызовов. 
  • Минус: Он знает только те символы, которые вы сами пропишете в словарь. Если в строке придет более редкая сущность (например, &copy; для знака © или числовой код вроде &#128512; для эмодзи), этот сниппет её проигнорирует. DOMParser же знает абсолютно все существующие HTML-сущности. 
Универсальный сниппет на RegExp 
Этот код автоматически декодирует абсолютно любые числовые коды (и &#1234;, и &#x1F600;), а также любые текстовые сущности, которые вы внесете в словарь namedEntities: 
javascript
const namedEntities = {
    '&amp;': '&', '&lt;': '<', '&gt;': '>', '&quot;': '"', '&#39;': "'",
    '&mdash;': '—',  // Длинное тире
    '&ndash;': '–',  // Среднее тире
    '&equiv;': '≡',  // Три тире (знак тождественности)
    // Сюда можно дописать любые другие редкие текстовые сущности
};

function decodeHtmlUniversal(str) {
    // Регулярное выражение ищет: текстовые сущности ИЛИ числовые коды (дес. и шестн.)
    return str.replace(/&[a-zA-Z0-9]+;|&#\d+;|&#[xX][a-fA-F0-9]+;/g, (match) => {
        // 1. Проверяем в словаре именованных сущностей
        if (namedEntities[match]) {
            return namedEntities[match];
        }
        
        // 2. Если это числовой код (начинается с &#)
        if (match.startsWith('&#')) {
            const isHex = match.charAt(3).toLowerCase() === 'x';
            // Вырезаем саму цифру в зависимости от формата (hex или dec)
            const code = isHex 
                ? parseInt(match.slice(4, -1), 16) 
                : parseInt(match.slice(2, -1), 10);
            
            // Превращаем код в реальный символ (работает и для эмодзи)
            return String.fromCodePoint(code);
        }
        
        // Если сущность неизвестна и не число — оставляем как есть
        return match;
    });
}

// Примеры работы:
console.log(decodeHtmlUniversal("Тире &mdash; и код &#128512;")); 
// Выведет: Тире — и код 😀
Что выбрать в итоге? 
  • Выбирайте DOMParser, если вы не хотите вручную вести список редких сущностей вроде знака тождественности &equiv;. Это «жирный», но абсолютно всеядный вариант. С вынесением объекта за пределы функции он станет достаточно быстрым для большинства задач. 
  • Выбирайте RegExp-сниппет, если функция вызывается критически часто (сотни раз в секунду), а набор нестандартных символов у вас ограничен и предсказуем. Вы просто один раз добавите их в объект namedEntities, и код будет летать.
