You said: В браузере Firefox есть команда openDialog, которая открывает новое окно. Что это за окно, в нём нет панелей, не работает контекстное меню.
Команда openDialog — это специфический метод JavaScript, используемый внутри движка Firefox. Он создает отдельное окно, предназначенное строго для диалогов (например: окно настроек, "О программе" или мастер установки). 
Почему оно такое? 
  • Отсутствие панелей: Окно является вспомогательным (popup), поэтому в нём нет адресной строки, кнопок «Назад/Вперед» и панели закладок. Это сделано для того, чтобы пользователь фокусировался только на самом диалоге, а не на интерфейсе браузера. 
  • Отсутствие контекстного меню: По умолчанию во многих типах таких окон отключены встроенные функции контекстного меню, чтобы предотвратить случайный уход со страницы диалога или открытие ненужных ссылок. 
  • Поведение: В отличие от стандартного метода window.open, он возвращает управление сразу, а для создания модального окна (блокирующего родительское окно) используется параметр modal. 
Эта функция является расширением интерфейса Window, специфичным для продуктов Mozilla (XUL). 
You said: обычный скрипт для сайта. хотелось бы знать, какие вообще параметры доступны для этого окна.
При использовании метода window.openDialog() обычным скриптом на сайте, движок Firefox обрабатывает его практически так же, как стандартный window.open(). Из соображений безопасности браузер блокирует большинство специфических системных параметров для обычных веб-страниц, оставляя их доступными только для внутренних расширений. 
Тем не менее для сайта доступны две группы параметров, которые передаются в третьем аргументе функции в виде строки features через запятую. 
1. Стандартные параметры управления окном (Замеры и Позиция) 
Эти параметры работают так же, как в window.open(). Браузеры уважают их, если они не нарушают границы экрана пользователя: 
  • width (или innerWidth) — ширина контентной области окна в пикселях.
  • height (или innerHeight) — высота контентной области окна в пикселях.
  • left (или screenX) — расстояние от левого края экрана до нового окна.
  • top (или screenY) — расстояние от верхнего края экрана до нового окна.
  • resizable (yes/no или 1/0) — разрешить ли пользователю вручную менять размер окна мышкой. 
2. Специфические параметры openDialog (Поведение) 
Главное отличие openDialog от open — это то, как он реагирует на параметры отображения интерфейса: 
  • modal — делает окно модальным. Пока пользователь не закроет этот диалог, он не сможет кликнуть или вернуться на родительскую вкладку сайта. Скрипт, вызвавший окно, «становится на паузу» на этой строчке. 
  • chrome — по умолчанию для openDialog этот флаг активен. Он заставляет окно открываться в минималистичном виде «диалоговой панели» — без адресной строки, панели закладок и контекстного меню. 
  • dependent — привязывает дочернее окно к родительскому. Если свернуть или закрыть основную вкладку сайта, диалог автоматически свернется или закроется вместе с ней. 
3. Передача аргументов внутрь окна (Уникальная фишка) 
Самая полезная возможность openDialog(), которая полноценно работает на обычных сайтах — это передача любых JavaScript-данных напрямую в новое окно без использования localStorage или postMessage. 
Вы можете передать их 4-м, 5-м и последующими аргументами: 
javascript
// Код на основной странице сайта:
let myData = { user: "Иван", role: "admin" };
window.openDialog("dialog.html", "MyDialog", "width=400,height=300,modal", myData);
Внутри открывшегося файла dialog.html эти данные мгновенно доступны через встроенный массив window.arguments: 
javascript
// Код внутри dialog.html:
console.log(window.arguments[0].user); // Выведет: Иван
⚠️ Важное ограничение безопасности 
Попытки прописать в строке features параметры вроде menubar=yes, location=yes или scrollbars=yes, чтобы вернуть контекстное меню или панели, внутри openDialog на обычном сайте будут проигнорированы браузером. Firefox жестко изолирует этот метод под интерфейсы «чистых диалогов».

Какие типы данных можно передавать? 
В openDialog() через четвертый и последующие аргументы можно передавать абсолютно любые типы данных JavaScript. Они передаются по ссылке («живьем»), а не копируются. 
Вы можете передать: 
  • Примитивы: String, Number, Boolean, null, undefined.
  • Сложные объекты и массивы: Любые JSON-подобные структуры.
  • Функции (Callback): Окно диалога может вызвать функцию, объявленную в коде родительской страницы.
  • DOM-элементы: Можно передать ссылку на конкретный <div> из родительского окна.
  • Ссылки на контекст: Чаще всего передают само родительское окно через ключевое слово window. 
Пример передачи функции и объекта: 
javascript
// Код на основном сайте:
function updateStatus(text) {
  console.log("Статус изменен на: " + text);
}

let config = { theme: "dark", userId: 42 };

// Передаем объект и функцию-колбэк
window.openDialog("dialog.html", "Dialog", "modal", config, updateStatus);
javascript
// Код внутри открывшегося dialog.html:
let parentConfig = window.arguments[0]; // { theme: "dark", userId: 42 }
let parentCallback = window.arguments[1]; // Функция updateStatus

// Прямо из диалога меняем данные и вызываем функцию родителя:
parentConfig.theme = "light"; 
parentCallback("Пользователь настроил тему!");
Если вы планируете передавать тяжелые данные или функции, учтите: при закрытии диалога ссылки могут «побиться», если родительский скрипт попытается обратиться к уничтоженному контексту. Хотите узнать, как правильно очищать память при такой передаче или как настроить общение окон через современные и безопасные методы? 
You said: Как?
Чтобы настроить безопасное общение между окнами и избежать утечек памяти, важно понимать одну деталь: при закрытии дочернего окна все ссылки на функции и объекты, которые были созданы внутри него, могут превратиться в «зомби-объекты» (dead wrappers) в родительском окне. 
Ниже приведены современные стандарты и правила очистки для старого openDialog. 
Способ 1. Современный стандарт (API postMessage) 
Это самый безопасный способ передачи данных между вкладками и окнами. Он работает асинхронно, не блокирует поток, не создает утечек памяти и защищен от уязвимостей, так как данные копируются (через алгоритм structured clone), а не передаются по прямой ссылке. 
Код на главной странице: 
javascript
// Открываем окно (работает и с open, и с openDialog)
const popup = window.open("dialog.html", "DialogPopup", "width=400,height=300");

// Слушаем сообщения от дочернего окна
window.addEventListener("message", (event) => {
  // ВАЖНО: Всегда проверяйте источник ради безопасности!
  if (event.origin !== window.location.origin) return;

  console.log("Данные от диалога:", event.data); // Получили объект { status: "success" }
});
Код внутри dialog.html: 
javascript
// Отправляем данные обратно родителю
function sendDataBack() {
  const dataToSend = { status: "success", timestamp: Date.now() };
  
  // window.opener — это автоматическая ссылка на родительское окно
  if (window.opener) {
    window.opener.postMessage(dataToSend, window.location.origin);
  }
}
Способ 2. Правильная очистка памяти в openDialog 
Если вы продолжаете использовать openDialog из-за его модальности (modal) и передаете туда живые функции-колбэки, необходимо вручную разрывать связи перед закрытием окна. В противном случае сборщик мусора (Garbage Collector) Firefox не сможет освободить оперативную память. 
Шаг 1. Передача данных из родителя: 
javascript
let myCallback = (result) => { console.log(result); };

// Передаем функцию третьим аргументом
window.openDialog("dialog.html", "Dialog", "modal", myCallback);
Шаг 2. Корректное завершение внутри dialog.html: 
javascript
// Функция, которая вызывается при нажатии на кнопку "Готово" в диалоге
function finishAndClose() {
  const callback = window.arguments[0];

  if (typeof callback === "function") {
    callback("Пользователь нажал ОК");
  }

  // ОБЯЗАТЕЛЬНО: Очищаем массив аргументов перед закрытием
  // Это стирает жесткие ссылки на функции родительского окна
  window.arguments = null; 

  // Теперь можно безопасно закрывать окно
  window.close();
}
Резюме: что выбрать? 
  • Если вам критически важно заблокировать интерфейс до ответа (модальность) в Firefox — используйте openDialog с обнулением window.arguments = null.
  • Если вы строите архитектуру на будущее, которая должна работать и в Chrome, и в Safari — используйте стандартный window.open в связке с window.postMessage.

You said: Это касается обычных окон, открытых через window.open() тоже?
Да, это касается обычных окон, открытых через window.open(), точно так же. 
Проблема «зомби-объектов» (в Firefox они технически называются Dead Wrappers) — это фундаментальное свойство безопасности браузерных движков, а не особенность конкретного метода openDialog. 
Почему это происходит в любом window.open()? 
Когда вы открываете обычное окно с помощью window.open(), браузер выделяет под него отдельный глобальный контекст (кучу памяти, среду выполнения JavaScript). 
Если вы передаете ссылку на объект или функцию из дочернего окна в родительское (например, через window.opener.myParentFunction = myChildFunction или сохраняя дочерний объект в родительскую переменную): 
  1. Пока окно открыто: Всё работает отлично. Родительское окно может свободно вызывать функции дочернего. 
  2. Окно закрывается: Контекст дочернего окна уничтожается, а вся его память принудительно очищается ради безопасности и предотвращения утечек. 
  3. Итог в родительском окне: Ссылка в родительском окне теперь указывает на «выжженную» область памяти. С этого момента любая попытка прочитать свойства этого объекта или вызвать дочернюю функцию вызовет жесткую ошибку безопасности: TypeError: can't access dead object. 
Как это выглядит на практике с window.open(): 
javascript
// КОД НА ГЛАВНОЙ СТРАНИЦЕ (РОДИТЕЛЬ)
let childDataLink = null;

const childWindow = window.open("child.html", "Child");

// Функция, которую вызовет дочернее окно для передачи данных
function receiveData(childObject) {
  childDataLink = childObject; // Сохранили ссылку на объект дочернего окна
}

// Позже, КОГДА ОКНО CHILD.HTML УЖЕ ЗАКРЫТО:
function tryToUseData() {
  // Крах скрипта! Ошибка: "can't access dead object"
  console.log(childDataLink.someProperty); 
}
Важное отличие: Направление передачи данных 
  • Безопасно (из Родителя в Дочернее): Если родитель передал свой объект в дочернее окно, и дочернее окно закрылось — родительский объект останется живым и невредимым в родительском окне. 
  • Опасно (из Дочернего в Родитель): Если родитель сохранил у себя ссылку на то, что родилось внутри дочернего окна, после закрытия этого окна ссылка превратится в "зомби". 
Именно поэтому современный стандарт window.postMessage() является единственным на 100% безопасным способом: он не передает ссылки «живьем», а клонирует сами данные.

Понимание природы этой ошибки — уже половина решения. Ошибка can't access dead object возникает потому, что Gecko (движок Firefox) физически уничтожает внутренний контекст (inner window) закрытого окна ради безопасности, а родительская страница пытается достучаться до его "призрака". 
Чтобы навсегда забыть об этой ошибке при работе с window.open(), используйте три проверенных архитектурных паттерна. 
Решение 1. Пассивное клонирование через JSON (Самое простое) 
Если дочернее окно передает только данные (без функций), никогда не сохраняйте ссылку на сам объект дочернего окна. Сделайте его моментальный снимок (клон) в памяти родителя. 
Как делать НЕ надо: 
javascript
// Внутри дочернего окна перед закрытием:
window.opener.saveData(myLocalObject); // Передает живую ссылку
Как делать НАДО (внутри дочернего окна): 
javascript
// Передаем глубокую копию. Родителю достанутся чистые данные, 
// которые никак не связаны с памятью закрывающегося окна.
window.opener.saveData(JSON.parse(JSON.stringify(myLocalObject)));
Альтернатива для современных браузеров: window.opener.saveData(structuredClone(myLocalObject)); 
Решение 2. Архитектура «Родитель — главный» (Для функций и колбэков) 
Вместо того чтобы дочернее окно передавало свои функции в родительское, переверните логику. Пусть родитель передает свои функции-слушатели дочернему окну. 
Когда дочернее окно закрывается, его ссылки на родителя уничтожаются, но родительские функции остаются живы, так как главное окно не закрывалось. 
javascript
// КОД В РОДИТЕЛЬСКОМ ОКНЕ
const child = window.open("child.html", "ChildPopup");

// Создаем функцию управления в родителе
window.onChildAction = function(data) {
  console.log("Данные получены в целости:", data);
};
javascript
// КОД В ДОЧЕРНЕМ ОКНЕ (child.html)
function sendAndClose() {
  if (window.opener && !window.opener.closed) {
    // Вызываем функцию родителя и передаем туда примитив или клон данных
    window.opener.onChildAction({ status: "ok" }); 
  }
  window.close();
}
Решение 3. Тотальная зачистка перед закрытием (Если код нельзя переписать) 
Если ваша текущая архитектура жестко завязана на то, что родитель хранит ссылки на объекты дочернего окна, вам необходимо принудительно обнулять эти ссылки в родительском скрипте в момент закрытия окна. 
Для этого подпишитесь на событие unload дочернего окна: 
javascript
// КОД В РОДИТЕЛЬСКОМ ОКНЕ
const child = window.open("child.html", "Popup");

// Ссылка, которая обычно превращается в "зомби"
let dangerousLink = null; 

// Функция для получения данных из дочернего окна
function setLink(obj) {
  dangerousLink = obj;
}

// Перехватываем момент уничтожения дочернего окна
child.addEventListener("unload", () => {
  // Не даем ссылке стать Dead Object — принудительно стираем её
  dangerousLink = null; 
  console.log("Дочернее окно закрылось, опасная ссылка очищена.");
});
