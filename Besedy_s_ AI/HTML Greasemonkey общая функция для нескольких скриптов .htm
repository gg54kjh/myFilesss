﹡Слушатели событий в окне Greasemonkey
﹡Общая функция для нескольких скриптов
greasemonkey window.addEventListener

1 час назад

В Greasemonkey (и других менеджерах пользовательских скриптов, таких как Tampermonkey) использование
window.addEventListener имеет свои особенности из-за работы скрипта в песочнице (изолированном окружении).
Основные моменты использования 
  • Контекст выполнения: По умолчанию скрипт выполняется в изолированном мире. window в вашем скрипте может не быть тем же самым объектом window (или unsafeWindow), который использует основная страница.
  • Событие load: Часто скрипты запускаются, когда DOM уже готов (document-end). Если вы добавите слушатель на window.load в этот момент, он может никогда не сработать, так как событие уже произошло.
    • Решение: Используйте директиву // @run-at document-start в метаданных скрипта, чтобы успеть перехватить событие загрузки.
Синтаксис и примеры 
Стандартный метод добавления слушателя:
javascript
window.addEventListener('click', function(event) {
    console.log('Клик по окну зафиксирован');
}, false);

Типичные ошибки 
  1. Немедленный вызов вместо ссылки: Передача myFunc() вместо myFunc приведет к выполнению функции сразу при регистрации, а не при наступлении события.
    • Неверно: window.addEventListener('load', myFunc(), false);
    • Верно: window.addEventListener('load', myFunc, false);
  2. Проблемы с this: Внутри обработчика this обычно указывает на объект, на который навешен слушатель (в данном случае window). При использовании стрелочных функций this берется из внешнего контекста.
Когда использовать window вместо document 
  • События окна: Для таких событий, как resize, scroll, load, unload, error или message, необходимо использовать именно window.
  • Глобальный перехват: window.addEventListener с параметром useCapture: true (третий аргумент) позволяет перехватывать события до того, как их обработает сама страница.
Взаимодействие со страницей (unsafeWindow) 
javascript
unsafeWindow.addEventListener('customPageEvent', function(e) {
    console.log('Событие со страницы получено:', e.detail);
});

Можно ли в Greasemonkey объявить функцию, общую для нескольких скриптов. а в скриптах вызывать эту функцию?

1 час назад

Да, это можно сделать несколькими способами, в зависимости от того, как именно вы хотите организовать код.
 
1. Использование директивы @require (Рекомендуемый способ) 
Это самый правильный и чистый метод. Вы выносите общие функции в отдельный .js файл (например, на GitHub, Pastebin или локальный сервер) и подключаете его в каждом скрипте. 
В библиотечном файле (common.js): 
javascript
function sayHello() {
    console.log("Привет из общей функции!");
}
Скопировано в буфер обмена
В заголовке каждого скрипта: 
javascript
// ==UserScript==
// @name         Мой скрипт
// @require      https://example.com
// ==/UserScript==

sayHello(); // Функция доступна напрямую
Скопировано в буфер обмена
Greasemonkey скачивает этот файл один раз при установке скрипта и хранит локально. 
2. Через объект unsafeWindow (Глобально на странице) 
Вы можете прикрепить функцию к объекту window самой страницы. Тогда она будет видна всем скриптам, работающим на этой вкладке. 
Скрипт-донор: 
javascript
unsafeWindow.mySharedLib = {
    log: function(msg) { console.log("Shared: " + msg); }
};
Скопировано в буфер обмена
Скрипт-потребитель: 
javascript
// Нужно дождаться, пока первый скрипт создаст объект
setTimeout(() => {
    if (unsafeWindow.mySharedLib) {
        unsafeWindow.mySharedLib.log("Работает!");
    }
}, 500);
Скопировано в буфер обмена
Минус: функция становится видна и самому сайту (его скриптам). 
3. Использование @grant GM_setValue / GM_addValueChangeListener 
Если нужно не просто вызвать функцию, а передать данные или сигнал между скриптами, можно использовать хранилище Greasemonkey. Один скрипт меняет значение, другой реагирует на это изменение. 
4. Библиотека в самом менеджере (только для Tampermonkey) 
Если вы когда-нибудь перейдете на Tampermonkey, там есть вкладка "Installed Libs", куда можно установить скрипт как библиотеку и подключать его по имени. 
Что выбрать? 
  • Если функции — это утилиты (хелперы), используйте @require.
  • Если скрипты должны взаимодействовать друг с другом в реальном времени, используйте unsafeWindow или события (CustomEvent).

      Важные нюансы: 
      • Кеширование: Greasemonkey скачивает файл по ссылке @require только один раз — при установке скрипта. Если вы обновите код на GitHub, пользователи (или вы сами на другом ПК) не получат обновления автоматически.
        • Решение: Чтобы принудительно обновить библиотеку, нужно переустановить скрипт или временно изменить версию в метаданных.
      • Приватность: Репозиторий на GitHub должен быть Public, иначе Greasemonkey не сможет получить доступ к файлу без авторизации.
      • Безопасность: Помните, что @require дает подключаемому файлу те же привилегии (доступ к GM_ функциям), что и у основного скрипта. Используйте только свой код или код из доверенных источников. 
      Как это выглядит в коде: 
      javascript
      // ==UserScript==
// @name         My Script
// @version      1.0
// @require      https://githubusercontent.com/utils.js
// @grant        none
// ==/UserScript==

// Теперь функции из utils.js доступны здесь
